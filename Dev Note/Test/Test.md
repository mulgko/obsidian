# Test

## 코드 테스트가 필요한 이유

### 1. 버그를 조기에 발견

코드를 수정할 때마다 기존 기능이 깨지는지(regression) 자동으로 확인할 수 있다. 직접 UI를 클릭해보지 않아도 된다.

### 2. 리팩토링에 대한 자신감

테스트가 있으면 코드 구조를 바꿔도 "기능은 그대로인가?"를 바로 검증할 수 있다. 테스트가 없으면 리팩토링이 무섭다.

### 3. 코드 동작의 문서화

```ts
it('음수를 입력하면 에러를 던진다', () => {
  expect(() => sqrt(-1)).toThrow()
})
```

테스트 코드 자체가 "이 함수가 어떻게 동작해야 하는가"를 설명한다.

### 4. 복잡한 엣지 케이스 처리

- 빈 배열, null, 음수, 매우 큰 숫자 등
- 일일이 UI에서 재현하기 어려운 상황을 자동화

### 5. CI/CD 파이프라인 연동

PR을 올렸을 때 테스트가 자동으로 돌아서, 깨진 코드가 main에 머지되는 걸 막는다.

## Vitest vs Jest 선택 기준

|  | **Jest** | **Vitest** |
|---|---|---|
| 생태계 | 더 오래됨, 레퍼런스 많음 | Vite 프로젝트에 최적 |
| 속도 | 상대적으로 느림 | ESM 네이티브, 훨씬 빠름 |
| 설정 | 별도 설정 필요 | `vite.config.ts` 재사용 |

- **Vite 기반 프로젝트 (React, Vue 등)** → Vitest
- **Next.js나 Node 백엔드** → Jest가 여전히 흔함
